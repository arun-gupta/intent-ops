# IntentOps  
**The control layer between intent and agent execution. Turn specs into execution plans with model routing and cost insight.**

---

## 1. Overview

IntentOps converts a **Spec-Driven Development (SDD) spec** into a **structured execution plan**, then executes it step-by-step using OpenHands.

The system provides:
- planning  
- task decomposition  
- model routing  
- token and cost estimation  
- controlled execution  
- plan vs execution observability  

IntentOps acts as the **control plane**.  
OpenHands acts as the **execution runtime**.

---

## 2. MVP Architecture

IntentOps MVP is built around five clearly defined layers:

- **SDD as the only MVP input**  
  IntentOps accepts a Spec-Driven Development spec as the sole input format.

- **SWE-bench as the MVP calibration layer**  
  Model routing decisions are informed by SWE-bench-style task characteristics and calibration signals.

- **OpenRouter as the model layer**  
  OpenRouter provides access to multiple models through a unified interface, enabling cost-aware routing.

- **OpenHands as the execution runtime**  
  OpenHands executes steps generated by IntentOps within a controlled environment.

- **Clear separation of planning vs execution**  
  IntentOps plans, routes, and estimates. OpenHands executes. Control remains with IntentOps.

---

## 3. Goals

- Transform SDD specs into structured execution plans  
- Recommend models per step  
- Estimate token usage and cost before execution  
- Execute steps in a controlled, observable loop  
- Track plan vs actual execution  
- Use SWE-bench as calibration for routing decisions

```text
Input → Validate → Normalize → Plan → Classify → Route → Estimate → Score → Execute → Report
```

## 8. Core Components

- provide rationale  

### 8.7 Estimator
- estimate tokens  

### 8.8 Cost Calculator
- estimate cost  

### 8.9 Scorer
- confidence  
- risk  
- readiness score  

### 8.10 Reporter
- JSON + Markdown output

## 11. Execution Layer (MVP)

IntentOps integrates with OpenHands.

---

### 11.1 Execution Model

```text
Plan → Step → Contract → OpenHands → Result → Evaluate → Next Step
```

---

### 11.2 Execution Contract

```json
{
  "step_id": "string",
  "goal": "string",
  "description": "string",
  "type": "string",
  "repo_context": {
    "files": [],
    "hints": []
  },
  "constraints": [],
  "success_criteria": [],
  "model": "string",
  "max_tokens": {
    "input": 0,
    "output": 0
  },
  "requires_human_review": false
}
```

---

### 11.3 OpenHands Adapter

- converts step → executable task  
- injects context and constraints  
- runs via OpenHands  
- captures outputs  

---

### 11.4 Execution Rules

- execute steps sequentially  
- respect dependencies  
- halt on failure or review gate  

---

### 11.5 Feedback Loop

After each step:
- evaluate success  
- compare expected vs actual  
- decide next action  

---

### 11.6 Observability

```json
{
  "step_id": "string",
  "status": "success | failed",
  "actual_input_tokens": 0,
  "actual_output_tokens": 0,
  "actual_cost_usd": 0.0,
  "retries": 0
}
```

---

### 11.7 Plan vs Execution Tracking

Track:
- estimated vs actual tokens  
- estimated vs actual cost  
- plan vs execution drift
